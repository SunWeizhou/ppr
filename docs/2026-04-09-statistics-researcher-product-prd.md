# arXiv Recommender 产品重构 PRD

状态：Draft  
日期：2026-04-09  
定位：统计学研究者的本地优先论文工作台

## 1. 背景

当前项目已经具备可用雏形：

- 能从 arXiv 拉取新论文并生成每日推荐。
- 能结合关键词、Zotero、喜欢/不喜欢、关注学者、顶刊追踪做个性化排序。
- 能通过本地 Web 页面完成浏览、反馈、搜索、设置和部分统计。

但从产品视角看，当前版本更像“功能集合”，还不是一个清晰、连续、可扩展的研究工作流产品。  
尤其对统计学研究者来说，真正核心的问题不是“我还能看到多少论文”，而是：

- 今天最值得读的 10-15 篇是什么。
- 哪几篇是真正偏统计理论/方法论，而不是泛 ML 热点。
- 我关注的学者、方向、期刊最近发生了什么变化。
- 我的阅读偏好有没有被系统持续学到。
- 我如何把推荐结果沉淀为自己的待读清单、研究线索和文献库。

因此，本 PRD 的目标不是做一个“更花哨的推荐页”，而是把项目重构为一个围绕统计研究流的产品。

## 2. 产品定位

### 2.1 产品定义

`arXiv Recommender` 是一个服务于统计学研究者的本地优先论文工作台，帮助用户每天以更低认知成本完成以下任务：

- 发现值得读的统计学新论文。
- 跟踪重点学者、期刊和研究方向。
- 维护个人研究兴趣画像。
- 沉淀待读、喜欢、收藏、标签、备注和导出记录。

### 2.2 目标用户

- 统计学博士生与博士后。
- 以统计理论、方法论、高维统计、非参数、推断为主的青年教师。
- 交叉做统计学习理论的研究者。
- 使用 Zotero 管理论文、希望每日筛选 arXiv 的个人研究者。

### 2.3 非目标用户

- 需要团队协作、批注分发、多人共享看板的实验室团队。
- 需要 SaaS 托管、云同步和跨设备统一账户的大众用户。
- 主要阅读工程应用、产品案例、代码仓库更新而非论文的用户。

## 3. 竞品分析与产品机会判断

### 3.1 样本范围

本轮外部参考对象按能力分组如下：

- 推荐与研究 feed：Semantic Scholar
- 研究容器与可视化工作流：ResearchRabbit、Litmaps
- 关系探索：Connected Papers
- 证据与质量判断：scite
- 问题驱动搜索：Elicit、Consensus
- 研究资产管理：Zotero、Paperpile
- 实体增强层：OpenAlex
- 开源同类参考：arxiv-sanity-preserver、zotero-arxiv-daily

### 3.2 外部产品已经验证的模式

- `Collection/Folder 是推荐种子`。成熟产品不会只依赖一个全局兴趣画像，而是允许用户围绕某个研究问题建立容器，并用该容器持续生成推荐。
- `Follow/Alert 是长期留存核心`。单次推荐只能解决“今天看什么”，长期价值来自持续提醒和跟踪对象命中。
- `Saved Search / Question Alert 是第二主入口`。研究者经常围绕问题追踪，而不是围绕单篇论文追踪。
- `Tags / Notes / Smart Collections 是研究资产层`。喜欢/收藏不足以形成研究工作流，必须支持沉淀和组织。
- `轻量 citation chaining 有价值`。从一篇种子论文继续看 related works、prior works、follow-up works 对研究推进很重要，但不应把首页做成 graph-first 产品。
- `实体与关系增强更适合后台能力`。作者、机构、期刊、主题、版本和开放获取信息应该作为增强层，而不是一开始就铺成重型前台页面。

### 3.3 本项目纳入的能力

| 能力 | 外部启发 | 为什么适合统计学研究者 | 为什么现在做 | 依赖对象、数据源与阶段 |
|------|----------|------------------------|--------------|------------------------|
| `ResearchCollection` 驱动推荐 | Semantic Scholar folder feed、ResearchRabbit collection | 统计研究经常围绕“一个问题线”推进，如 conformal under shift、finite-sample inference | 当前产品只有全局兴趣画像，缺少研究问题级容器，是主线短板 | 依赖 `ResearchCollection`、`PaperIdentity`、Zotero/arXiv；Phase 1 |
| `SavedSearch` / 问题订阅 | Elicit、Consensus、Zotero feeds | 研究者会长期跟踪一个问题或术语组合，而不是只跟踪作者 | 当前搜索是一次性动作，缺少长期价值 | 依赖 `SavedSearch`、查询规范化、arXiv/OpenAlex 增强；Phase 1 |
| 统一 `Follow System` | ResearchRabbit、Litmaps、Semantic Scholar alerts | 学者、期刊、主题本质上都是长期关注对象 | 当前追踪分散在多个页面和数据文件里 | 依赖 `FollowTarget`、多源规范化；Phase 1 |
| `Inbox -> Queue -> Archive` 研究队列 | Zotero/Paperpile 的研究资产管理范式 | 统计研究者需要区分值得精读、暂存、归档，而不是只点赞 | 当前喜欢/收藏/历史关系太弱 | 依赖 `ReadingQueueItem`、事件流；Phase 1 |
| Zotero 导入导出工作流 | ResearchRabbit Zotero importer、Zotero 本身 | Zotero 已是目标用户的核心工具，接入成本低 | 当前已有 Zotero 输入基础，最适合扩展为双向工作流 | 依赖 Zotero connector/导出能力、`export_to_zotero` 事件；Phase 1 |
| 轻量 `related/prior/follow-up` 探索 | Connected Papers、Litmaps | 统计研究需要顺着方法脉络补前作和后续工作 | 当前系统偏筛列表，缺少推进研究问题的路径 | 依赖 `PaperIdentity`、引用/相似关系增强；Phase 2 |
| 系统状态页与运行状态模型 | 成熟研究工具的 sync/status 反馈 | 本地优先产品必须降低“到底是没更新还是坏了”的不确定性 | 当前运行状态依赖缓存文件探测，不可解释 | 依赖 `JobRun`、任务生命周期；Phase 1 |
| OpenAlex 作为后续实体增强层 | OpenAlex | 可补作者、机构、主题、版本、开放获取、去重关系 | 这类能力适合作为增强层，不应阻塞主流程改版 | 依赖 source adapter、实体规范化；Phase 3 |

### 3.4 本项目明确不纳入的能力

| 能力 | 外部启发 | 为什么当前不做 | 对定位的风险 | 后续处理 |
|------|----------|----------------|--------------|----------|
| 图谱优先首页或全局 citation graph | Connected Papers、Litmaps | 统计研究者的主任务仍是“今天看什么”，不是先看全局图谱 | 会把首页从工作台做成探索器，稀释主线 | 明确不做首页核心，仅保留轻量单篇扩展 |
| 通用 AI 问答 / 全文聊天 / agent 化综述 | 各类 AI 研究助手 | 与当前“每日研究工作台”目标不一致，且实现复杂度高 | 会把产品误导成通用学术助手 | 不纳入近期范围 |
| 团队协作、共享知识库、云同步账号体系 | SaaS 文献平台 | 当前目标用户是个人统计研究者，本地优先是明确定位 | 会显著放大工程复杂度和产品表面积 | 不纳入近期范围 |
| scite 级别 citation stance 作为 v1 核心 | scite | 证据网络有价值，但并非当前主线瓶颈 | 会提前放大数据依赖与授权问题 | 仅保留为 Phase 3+ 探索项 |
| 大而全 systematic review 主流程 | Elicit、Consensus | 统计研究者的主任务是持续追踪和筛选，不是系统综述自动化 | 会让产品偏离每日工作台定位 | 不纳入近期范围 |
| 跨学科大召回 | 通用学术搜索引擎 | 当前产品价值来自“统计优先”的筛选与解释 | 会稀释排序、过滤和解释精度 | 不做近期方向扩张 |

## 4. 当前产品诊断

### 4.1 需要优化的地方

| 领域 | 当前问题 | 用户影响 | 优先级 |
|------|----------|----------|--------|
| Onboarding | 首次使用仍依赖配置文件、路径、环境理解 | 新用户启动成本高，容易在第一步流失 | P0 |
| 导航结构 | 今日推荐、搜索、学者、顶刊、喜欢、统计、设置是并列页面，但缺少主任务线 | 用户知道“功能在哪”，但不知道“今天先做什么” | P0 |
| 推荐解释 | 解释主要是关键词命中和少量理由，缺乏 why this、why above、why hidden 的完整闭环 | 用户不易建立信任，也难以手动校准系统 | P0 |
| 反馈动作 | 当前以喜欢/不喜欢为主，动作粒度太粗 | 无法区分“值得精读”“先放待读”“只是今天不看” | P0 |
| 结果组织 | 推荐结果、收藏、喜欢、历史记录分散 | 用户形成不了稳定的阅读工作流 | P0 |
| 学者与顶刊功能 | 有功能，但与主推荐流分离 | 关注关系无法真正影响“今日信息流”体验 | P1 |
| 统计页 | 更像数据展示，不像产品反馈面板 | 无法指导用户调整偏好，也无法验证系统是否学会了用户 | P1 |
| 搜索 | 独立存在，无法保存搜索、订阅搜索或转化为长期兴趣 | 短期可用，长期价值低 | P1 |
| 系统状态 | 缺少明确的抓取状态、失败原因、最近更新时间、Zotero 连接状态 | 出问题时用户不知道是没论文、没配置好还是程序坏了 | P1 |
| 文档一致性 | README、实际配置结构和代码实现存在漂移 | 影响理解和后续开源、交接、协作 | P1 |

### 4.2 需要重构的地方

| 领域 | 当前问题 | 目标状态 | 优先级 |
|------|----------|----------|--------|
| 后端结构 | 推荐、抓取、缓存、HTML 生成耦合严重 | 拆分为 ingestion、ranking、profile、serving、jobs 五层 | P0 |
| Web 层 | 大量 HTML/CSS 通过 Python 字符串拼接 | 改为模板层或独立前端组件层 | P0 |
| 数据存储 | 用户状态散落在多个 JSON 和 cache 文件中 | 统一到本地 SQLite 或统一数据仓层，JSON 用于导出备份 | P0 |
| 配置管理 | 配置 schema 正在迁移但仍有历史包袱 | 明确单一配置源与版本迁移机制 | P0 |
| 跨平台能力 | 仍存在硬编码路径和 Windows 假设 | 完全相对路径化与平台无关化 | P0 |
| 推荐逻辑 | 候选召回、打分、反馈学习混在一起 | 形成可测试、可评估的推荐流水线 | P1 |
| 可观测性 | 以日志为主，缺少结构化运行状态 | 提供 run status、error taxonomy、数据源健康检查 | P1 |
| 评估框架 | 没有离线评估与线上指标定义 | 能比较不同权重/策略版本的效果 | P1 |

### 4.3 需要重新设计的地方

| 模块 | 为什么必须重设计 | 新方向 | 优先级 |
|------|------------------|--------|--------|
| 核心信息架构 | 当前是“功能页集合”，不是“研究流程产品” | 以“今日收件箱 -> 研究队列 -> 跟踪 -> 洞察”为主线 | P0 |
| 个性化模型 | 当前偏规则驱动，反馈学习较浅 | 设计可解释、可调参、可复用的兴趣画像系统 | P0 |
| 跟踪体系 | 学者、期刊、关键词分散维护 | 统一为 `FollowTarget + ResearchCollection + SavedSearch` | P0 |
| 阅读工作流 | 缺少待读、归档、导出、标签、精读状态 | 重构为 `Research Queue + Research Assets` | P0 |
| 价值表达 | 现在更像“推荐器”，不是“研究助手” | 重构为统计研究者每日研究工作台 | P1 |

## 5. 产品目标

### 5.1 业务目标

- 提升统计研究者筛选新论文的效率。
- 提升推荐结果的信任感、可解释性和可控性。
- 建立稳定的本地化个人研究画像。
- 把“看推荐”升级为“持续研究工作流”。

### 5.2 用户目标

- 每天 5 分钟内看到真正值得读的候选论文。
- 能快速区分“现在读”“之后读”“忽略”“长期关注”。
- 能持续追踪统计学重要学者、顶刊和主题变化。
- 不需要频繁手动改配置文件也能让系统越用越准。

### 5.3 非目标

- 不做通用学术搜索引擎。
- 不做论文全文问答平台。
- 不做多人协作型研究管理系统。
- 不强制依赖云端账户、远程数据库或在线服务。

## 6. 核心设计原则

- 统计优先：产品默认服务于统计理论、方法与统计学习方向，而非泛 ML 热点。
- 本地优先：用户数据、本地缓存、个人偏好默认留在本机。
- 少而准：每天控制候选数量，优先“值得读”而不是“尽量全”。
- 可解释：每条推荐都必须告诉用户为什么出现、为什么排前、为什么被压后。
- 可控：用户必须能方便地加权、降权、关注、忽略。
- 容器优先：推荐围绕 `ResearchCollection` 和 `SavedSearch` 等研究容器组织，而不是只围绕一次性列表。
- 一条主线：首页永远围绕“今天该读什么”展开，其它能力为主线服务。

## 7. 目标产品方案

### 7.1 一级对象与承载页面

本产品不再只按页面设计，而按以下一级对象组织：

| 一级对象 | 定义 | 主要承载页面 |
|----------|------|--------------|
| `FollowTarget` | 用户长期关注的作者、期刊、会议、主题、arXiv 类别 | 关注动态、今日收件箱 |
| `ResearchCollection` | 围绕一个研究问题或子方向建立的研究容器，可包含种子论文、标签、备注 | 探索搜索、研究队列、今日收件箱 |
| `SavedSearch` | 可重复运行的查询与订阅，围绕问题长期追踪新论文 | 探索搜索、关注动态、今日收件箱 |
| `ReadingQueueItem` | 论文在研究队列中的当前状态投影 | 研究队列与资产 |
| `ResearchAsset` | 与论文绑定的标签、备注、导出记录和组织信息 | 研究队列与资产 |
| `RecommendationRun` | 一次推荐生成的候选、排序、解释与结果集合 | 今日收件箱、研究洞察 |
| `JobRun` | 一次抓取、生成、刷新、导出等任务的运行状态 | 设置与系统状态 |

### 7.2 双主线信息架构

产品明确采用双主线：

1. `Daily Inbox`
   解决“今天看什么”。
2. `Question Tracking`
   解决“我长期在追什么问题”。

对应页面为：

1. 今日收件箱
2. 关注动态
3. 探索搜索
4. 研究队列与资产
5. 研究洞察
6. 设置与系统状态

### 7.3 各模块定义

#### 今日收件箱

产品定位：用户每天打开后最先进入的页面。

供稿来源明确为：

- 全局兴趣画像命中
- `FollowTarget` 命中
- `ResearchCollection` 种子扩展命中
- `SavedSearch` 命中

核心内容：

- 今日 Top 12-20 篇论文。
- 每篇论文显示来源、主题、理由、分数、发布日期、版本信息。
- 支持一键标记为“值得精读 / 稍后再看 / 忽略 / 关注作者 / 加入 Zotero”。
- 明确展示 why this、why above、why hidden 的入口。

#### 关注动态

产品定位：统一的跟踪中心，而不是分散的“学者页”“顶刊页”。

核心内容：

- 我关注的学者最新论文。
- 我关注的期刊/会议最新更新。
- 我关注的关键词/主题相关新增论文。
- 我保存的问题订阅命中的新增论文。
- 支持按来源过滤，但默认以 `FollowTarget` 聚合展示。

#### 探索搜索

产品定位：为临时搜索和长期兴趣发现服务。

核心内容：

- 支持关键词搜索、作者搜索、主题搜索。
- 支持把搜索保存为 `SavedSearch`。
- 支持把搜索结果转为 `ResearchCollection`。
- 支持从单篇论文继续扩展 related works、prior works、follow-up works。

#### 研究队列与资产

产品定位：把推荐结果转化为个人研究资产。

核心内容：

- `Research Queue`：Inbox、Skim Later、Deep Read、Saved、Archived。
- `Research Assets`：tags、notes、smart collections、导出记录。
- 区分“研究状态”和“偏好动作”，避免喜欢、收藏、待读混在一起。

#### 研究洞察

产品定位：帮助用户理解自己的研究输入，而不是只看数量。

核心内容：

- 最近 7/30 天偏好主题变化。
- 最常喜欢的作者、关键词、来源。
- 推荐命中率与纠偏效果。
- 新兴主题、活跃学者、顶刊趋势。
- `SavedSearch` 和 `FollowTarget` 的命中表现。

#### 设置与系统状态

产品定位：配置中心 + 健康检查中心。

核心内容：

- 研究方向模板选择。
- 核心/次级/排斥关键词管理。
- Zotero 状态、最近同步时间、最近生成状态。
- 数据源开关、每日推荐数量、抓取频率。
- `JobRun` 任务状态、失败原因、降级运行状态。

## 8. 关键需求

### 8.1 Onboarding 与冷启动

#### 需求

- 新用户应在 10 分钟内完成首次可用。
- 不应要求用户手写 JSON。
- 应支持“无 Zotero 模式”和“有 Zotero 模式”。
- 应支持统计学方向模板快速导入。

#### 方案

- 提供引导式配置向导。
- 提供统计学模板：
  - 统计学习理论
  - 高维统计
  - 非参数估计
  - Conformal Prediction
  - 贝叶斯与推断
  - 因果与实验设计
- Zotero 接入后自动生成初始兴趣画像。
- 配置完成后，直接展示一版可预览的今日收件箱。

### 8.2 今日收件箱

#### 需求

- 默认展示最相关的 12-20 篇。
- 每篇都必须给出理由卡片。
- 必须支持快速动作而非只点喜欢/不喜欢。
- 必须限制同作者、同方向、同来源的刷屏。

#### 页面结构

- 顶部概览：今日共抓取多少候选，最终筛出多少篇。
- 筛选器：理论优先、近 3 天、仅关注学者、仅顶刊来源。
- 论文卡片：
  - 标题、作者、来源、时间。
  - 主理由：为什么推荐。
  - 次理由：为什么排前。
  - 控制入口：如何调整类似论文。
  - 快速动作按钮。

#### 新动作设计

- 值得精读
- 加入待读
- 忽略本篇
- 忽略该主题
- 关注作者
- 关注期刊/会议
- 导出到 Zotero

### 8.3 `ResearchCollection` 与 `SavedSearch`

#### 需求

- 用户必须能围绕一个研究问题建立长期容器，而不是只依赖全局兴趣画像。
- 搜索结果必须能沉淀为可持续维护的对象。

#### `ResearchCollection` 定义

- 表示一个研究问题、方法分支或子方向。
- 可包含种子论文、手动加入论文、标签、备注。
- 可作为推荐种子、跟踪对象和导出容器。

#### `SavedSearch` 定义

- 表示一个可重复执行的查询和订阅。
- 以问题为中心，而不是以单篇论文为中心。
- 可单独命中提醒，也可作为收件箱供稿来源。

### 8.4 Follow System

#### 需求

- 用户不应分别维护“我的学者”“顶刊列表”“关键词订阅”。
- 所有长期兴趣都应统一进入一个关注系统。

#### 关注对象

- 作者
- 期刊
- 会议
- 主题
- arXiv 类别
- `SavedSearch`

#### 产品行为

- 关注对象可以影响候选召回与重排。
- 关注对象可以单独产生提醒。
- 关注动态默认汇总到“关注动态”页，而不是散落成多个孤岛页面。
- `ResearchCollection` 可以生成多个 `FollowTarget` 和一个或多个 `SavedSearch`。

### 8.5 研究队列与研究资产

#### 需求

- 用户需要把推荐结果沉淀为长期可管理资产。
- 喜欢不等于待读，收藏不等于读完。

#### 状态设计

- Inbox
- Skim Later
- Deep Read
- Saved
- Archived

#### 资产设计

- tags
- notes
- smart collections
- Zotero 导出记录

#### 说明

- 该状态机用于替代当前“喜欢/收藏/历史”之间的弱关系。
- 队列状态是当前投影，交互行为保留为事件流。

### 8.6 推荐解释与控制

#### 需求

- 用户必须知道推荐的主要原因。
- 用户必须能控制推荐偏差。

#### 解释模型

- `why this`：为什么这篇论文会出现。
- `why above`：为什么这篇比类似论文排得更前。
- `why hidden`：为什么某类论文被压后或没进榜。

#### 解释维度

- 命中的统计主题。
- 与 Zotero 语义相似。
- 来自关注学者。
- 来自顶刊/顶会。
- 理论深度高。
- 新近度高。
- 来源可信度与版本信息。

#### 控制维度

- 提升某一主题权重。
- 降低某一主题权重。
- 屏蔽某位作者。
- 限制工程类或应用类论文比例。
- 控制 theory、methodology、application 倾向。

### 8.7 研究问题扩展路径

#### 需求

- 系统不能只停留在“给列表”，还要帮助用户推进研究问题。

#### 主流程

- 从单篇论文进入 related works。
- 从单篇论文进入 prior works。
- 从单篇论文进入 follow-up works。
- 允许一键转成 `SavedSearch` 或 `ResearchCollection`。

#### 边界

- 不做全局图谱首页。
- 不做复杂可视化工作台作为首页核心。

### 8.8 研究洞察

#### 需求

- 洞察页必须回答“系统是否真的理解我”。
- 不仅展示阅读量，还要展示偏好变化、命中效果和纠偏效果。

#### 关键模块

- 最近偏好主题分布。
- 最近 30 天高频喜欢作者。
- 推荐命中率：
  - Top 10 被喜欢、加入待读、进入 Deep Read 的比例。
  - `FollowTarget` 命中率。
  - `SavedSearch` 命中率。
  - 顶刊命中率。
- 新兴趋势：
  - 最近高频出现的统计主题。
  - 活跃学者与机构。

## 9. 推荐系统重设计

### 9.1 目标

将当前推荐能力重构为“可解释的多阶段排序系统”，而不是单层混合打分。

### 9.2 新推荐架构

#### Stage 1：候选召回

候选来自以下来源：

- arXiv 类别召回。
- 用户关注主题的主动搜索。
- `FollowTarget` 命中召回。
- `SavedSearch` 命中召回。
- `ResearchCollection` 种子扩展召回。
- 顶刊或会议新增召回。
- 最近未读但高相关的补充召回。

#### 召回池配额

- 首页默认展示 12-20 篇。
- `arXiv 新论文` 是主召回池。
- `FollowTarget` 命中保证最少展示位，避免被纯相关性稀释。
- `SavedSearch` 命中可单独进入首页候选池。
- `顶刊/期刊新增` 不与 arXiv 争同一实时池，采用并行展示或固定插槽。
- `探索位` 保留固定比例，避免过拟合。
- `arXiv 快、顶刊慢` 的产品行为明确为：
  - arXiv 主导首页日更节奏。
  - 顶刊更新以“关注动态”和“质量增强信号”为主。
  - 顶刊新增可进入首页固定插槽，但不主导首页刷新节奏。

#### Stage 2：特征抽取

每篇论文提取统一特征：

- 主题相关性。
- 统计理论深度。
- 与 Zotero 相似度。
- 学者或期刊亲和度。
- 新近度。
- 新颖性与去重度。
- 来源可信度。
- 用户负反馈风险。
- 版本成熟度。

#### 统计学专用重排准则

排序必须显式识别以下信号：

- theory、methodology、application 类型。
- finite-sample、asymptotic 倾向。
- 是否包含 theorem、proof、bound、guarantee 等核心理论信号。
- 是否只是泛 ML 热点包装而非统计问题本体。
- 是否已存在期刊终版或后续版本。
- 来源可信度与版本成熟度。

各信号的角色锁死如下：

- `统计主题命中`、`理论深度`、`FollowTarget`、`SavedSearch`、`ResearchCollection` 作为排序主信号。
- `来源可信度`、`版本成熟度` 既参与排序加权，也参与解释展示。
- `是否为泛 ML 热点包装` 作为降权信号。
- `顶刊新增` 既可加权，也可作为单独展示维度。

#### Stage 3：重排与多样性控制

重排策略需要满足：

- Top 10 中控制单作者数量上限。
- 避免同一主题连续刷屏。
- 保留固定探索位，避免过拟合。
- 允许顶刊、关注对象和问题订阅在首页并存，而不是互相覆盖。

#### Stage 4：解释生成

每篇论文生成：

- 一个主理由。
- 两到三个辅助理由。
- 一个用户可调的控制入口。

解释输出必须覆盖：

- `why this`
- `why above`
- `why hidden`

### 9.3 反馈学习重设计

当前问题是反馈更像“即时状态”，而不是可复用的用户行为数据。  
新版本需要基于统一交互事件做长期学习。

#### 事件语义

以下事件属于统一事件流：

- `impression`
- `open_paper`
- `save_for_later`
- `deep_read`
- `like`
- `dislike`
- `follow_author`
- `follow_topic`
- `follow_search`
- `export_to_zotero`

以下状态属于当前状态投影：

- `Inbox`
- `Skim Later`
- `Deep Read`
- `Saved`
- `Archived`

语义锁定如下：

- `ignore topic` 生成持久规则。
- `like` 独立于队列状态。
- `export_to_zotero` 记录事件，不自动改变阅读状态。
- `impression` 是离线评估与解释纠偏的基础事件。

#### 学习逻辑

- 显式反馈优先于隐式反馈。
- 长期反馈应跨天生效，而不是只在当天候选集合内生效。
- 事件应支持时间衰减和作用范围定义。
- 用户应能看到“系统最近学到了什么”。

## 10. 数据模型建议

建议引入统一实体模型：

- `Paper`
- `PaperIdentity`
- `PaperVersion`
- `PaperSource`
- `UserProfile`
- `InterestKeyword`
- `FollowTarget`
- `ResearchCollection`
- `SavedSearch`
- `InteractionEvent`
- `ImpressionEvent`
- `RecommendationCandidate`
- `RecommendationRun`
- `ReadingQueueItem`
- `ResearchAsset`

## 11. 系统约束

### 11.1 数据层约束

- SQLite 从 Phase 1 起是唯一写路径。
- JSON 仅用于导入、导出与一次性迁移。
- 现有 `user_feedback`、`favorite_papers`、`recommendation_history` 必须有一次性迁移方案与 schema version。
- `PaperIdentity` 负责规范化论文身份，`PaperVersion` 负责版本映射。

### 11.2 多源接入约束

- 多源接入必须先做 `paper / author / venue / topic / search` 规范化，再谈 follow、alert、洞察。
- `OpenAlex` 等增强层只在统一实体层之上接入，不能绕过规范化直接写前台逻辑。
- 顶刊、arXiv、Zotero、SavedSearch 命中必须回到同一候选模型里比较与解释。

### 11.3 任务状态约束

- 任务模型必须有 `run_id`。
- 生命周期必须覆盖：
  - `queued`
  - `running`
  - `succeeded`
  - `degraded`
  - `failed`
- 系统状态页必须依赖 `JobRun`，不能继续直接探测缓存文件判断是否成功。

### 11.4 最小接口合同

PRD 级别最小接口合同如下，不在本文件中定义具体 JSON schema：

- 获取今日收件箱
- 获取关注动态
- 保存搜索
- 管理 `ResearchCollection`
- 更新研究队列状态
- 查询 `job/status`

### 11.5 评估门槛

- 推荐系统重构必须同时具备线上产品指标和离线评估指标。
- 任何新排序策略上线前，必须能在固定候选快照上进行回归比较。
- 解释层必须能够被用户动作反向验证，而不只是静态文案。

## 14. 成功指标

### 14.1 激活指标

- 首次配置完成率 > 70%
- 首次成功生成推荐时间 < 10 分钟

### 14.2 推荐质量指标

- Top 10 推荐中被标记为“值得精读”或“加入待读”的比例 > 35%
- 用户主动调整关键词或关注对象后，次日命中率提升可感知
- `FollowTarget` 命中率持续提升
- `SavedSearch` 命中率持续提升

### 14.3 使用行为指标

- 7 日内至少二次回访率 > 40%
- 每日打开后 3 分钟内完成至少一个动作的比例 > 60%
- `SavedSearch` 7 日和 30 日留存
- `Inbox -> Deep Read` 转化率
- Zotero 导出成功率
- 推荐解释触发后的调权或纠偏率

### 14.4 离线评估指标

- `precision@k`
- 召回覆盖率
- 多样性与去重率
- 延迟预算
- 解释命中率

### 14.5 稳定性指标

- 每日任务成功率 > 95%
- 数据源抓取失败可解释率 100%
- `JobRun` 状态与用户可见状态一致

## 15. 风险与约束

- 统计学方向本身论文量低于泛 ML，不能用“大而全”思路设计召回。
- 顶刊更新慢，arXiv 快，两类节奏差异必须通过并行展示与固定插槽解决，而不是简单混排。
- 本地优先意味着安装与环境问题不会完全消失，只能被向导和诊断工具吸收。
- 若过度依赖关键词，系统会对新兴主题不敏感；若过度依赖语义相似，又容易变成黑盒。
- 若过早做图谱首页、通用 AI 问答、团队协作，会显著偏离当前定位并拉高工程复杂度。

## 16. 本 PRD 的产品结论

这个项目最应该做的，不是继续给现有页面加功能，而是先把产品重心从“展示推荐结果”改成“服务统计学研究者的每日研究工作流”。

最关键的四件事：

1. 重做产品主线：把首页改成真正的今日收件箱。
2. 引入研究容器：以 `ResearchCollection` 和 `SavedSearch` 承接问题驱动工作流。
3. 重做状态模型：把喜欢、收藏、历史改造成研究队列与研究资产。
4. 重做工程底座：把规则、数据、页面和任务调度拆开，建立统一实体层、任务状态层和可评估的推荐架构。

如果这几件事不做，后续再叠加趋势分析、更多推荐信号、更多页面，产品复杂度会上升，但用户感知价值不会同步增长。
