Phase 2：把 Agent Confirmation Flow 真正做成闭环

这是这一轮最重要的中型任务。

2.1 当前问题

前端已经做了：

confirmation token state；
Confirm / Cancel UI；
Confirm 时向 API 发送 confirmation_token。

但后端：

route 不读取 confirmation_token；
AgentService.handle_message() 不接收 token；
不保存 pending action；
confirm 之后不会执行原始动作。
2.2 目标行为

对于敏感行为：

用户：

delete all archived agent sessions

Agent：

This action requires confirmation before I run it.
UI 展示 Confirm / Cancel。
用户点击 Confirm。
后端真正执行刚才待确认的动作。

Agent 返回：

Confirmed. I deleted ...
2.3 推荐实现方案
数据层：新增 pending confirmation 存储

建议在 SQLite 中新增表：

agent_pending_confirmations

字段建议：

token TEXT PRIMARY KEY
session_id TEXT NOT NULL
message TEXT NOT NULL
plan_json TEXT NOT NULL
page_context_json TEXT NOT NULL
created_at TEXT NOT NULL
expires_at TEXT NOT NULL
status TEXT NOT NULL DEFAULT 'pending'
服务层：AgentService 增强
修改签名
handle_message(
    message: str,
    session_id: Optional[str] = None,
    page_context: Optional[dict] = None,
    confirmation_token: Optional[str] = None,
)
分支逻辑
如果 confirmation_token 存在：
load pending confirmation；
校验 session；
校验未过期；
执行之前存储的 plan；
更新状态为 confirmed / consumed；
返回执行结果。
如果当前请求触发 destructive：
不执行；
生成 token；
持久化 pending confirmation；
返回 requires_confirmation=True 和 token。
2.4 Route 层修改

app/routes/api/agent.py 需要读取：

confirmation_token = data.get("confirmation_token")

并传给 service。

2.5 前端优化建议

现有 UI 可以保留，但建议：

Confirm 文案显示具体动作，而不是笼统的：

This action requires your confirmation.

例如：

Confirm deleting 12 archived sessions?

Cancel 后最好追加一条轻量 assistant/system message：

Action cancelled.

2.6 验收

至少补以下测试：

destructive request returns requires_confirmation=True
confirm with valid token executes action
invalid token rejected
expired token rejected
token cannot be reused
cancel does not execute action

Phase 6：Recommendations 更可信，不要“假个性化”

6.2 推荐过滤最好以最终 composite score 为准

建议

composite score 过低的不展示；

这样可信度更强。

Phase 7：Entity 系统做“数据链路补强”

这部分我建议 Agent 做，因为现在 UI 做出来了，但外部 ID 链路还不够完整。

7.1 Venue metadata sync 目前大概率经常拿不到数据
当前问题

EntityService.extract_entities_from_results() 希望拿：

paper_ext.get("openalex_source")

来创建 venue entity 的 OpenAlex source ID。

但 normalize_openalex_paper() 当前的 external_ids 只存：

doi
openalex work id

并没有存 source id。

结果

很多自动生成的 journal/conference entity 没有 source external id，Sync Metadata 无法真正 fetch。

建议

在 OpenAlex normalize 时新增：

external_ids["openalex_source"] = source_id

其中 source_id 来自：

primary_location.source.id

并适当去掉 https://openalex.org/ 前缀。

7.2 Scholar entity 目前多半只有名字，没有外部 ID
当前问题

Entity 提取时：

entity = self.get_or_create(entity_type="scholar", name=name)

只是用作者名建 scholar entity。

而 sync_metadata() 对 scholar 依赖：

semantic_scholar author id

结果

很多 scholar entity 即使显示了 Profile，也没法真正同步作者元数据。

建议

在 paper normalization 阶段保留作者 ID 信息：

Semantic Scholar

保留：

authorId
name
OpenAlex

保留：

author id
display_name

可以新增字段：

"author_entities": [
  {"name": "...", "semantic_scholar": "...", "openalex": "..."}
]

然后 EntityService 优先使用外部 ID 创建 scholar entity。

7.3 控制搜索结果自动抽 entity 的数量

现在每次搜索可能给所有作者都建 scholar entity。

风险
entity 表快速膨胀；
很多弱质量作者节点；
Watch / Entity UX 变脏。
建议

短期策略：

每篇 paper 只抽前 3 个作者；
每次 search 最多创建 N 个 scholar entity；
或仅对：
selected paper
saved paper
top-ranked papers
进行 entity extraction。

这项可以先做保守限流。
